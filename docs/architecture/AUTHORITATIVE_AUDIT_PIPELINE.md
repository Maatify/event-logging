# Authoritative Audit Pipeline

**Status:** CANONICAL
**Scope:** Details the specific fail-closed architecture, storage mechanics, and future consumer strategies for the AuthoritativeAudit domain.

---

## 1. Domain Intent

The `AuthoritativeAudit` domain is distinct from the other five logging domains. It is designed specifically for **governance, security posture, compliance, and strict state transitions**. It is the absolute source of truth for critical system actions.

## 2. Fail-Closed Guarantees

Unlike all other domains which are fail-open, AuthoritativeAudit is strictly **fail-closed**.
- The `AuthoritativeAuditRecorder` will allow exceptions (such as `SystemMaatifyException`) generated by the storage layer to propagate to the caller.
- This ensures that if the audit log cannot be written, the host application's containing transaction or operation fails and rolls back. An action must not occur if it cannot be audited.
- There is no fallback to PSR-3 logging; failure to store means failure to execute.

## 3. Storage Mechanics

The domain operates with two tables:

1. **`maa_event_logging_authoritative_audit_outbox`**
   - The authoritative source of truth.
   - Written to synchronously by the `AuthoritativeAuditRecorder` within the host application's primary database transaction.
   - Ensures atomicity between the business action and the audit record.

2. **`maa_event_logging_authoritative_audit_log`**
   - The primary schema/read model.
   - Intended for querying, reporting, and immutable historical reference.

## 4. Pipeline & Materialization (Deferred)

The complete pipeline design contemplates an asynchronous consumer (materializer) that reads from the outbox and moves records to the audit log table, potentially across database boundaries (e.g., from a high-transaction OLTP database to a dedicated audit data warehouse).

**Important:** The implementation of outbox consumers, materializers, and dead-letter semantics for handling consumer failures is **deferred** and is not part of the v1.0.0 package baseline. The package provides the schema and the outbox repository for writing, but the host application is responsible for orchestrating the materialization process if desired.
